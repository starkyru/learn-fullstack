You produce a **personalized** learning path through `learn-fullstack` and save it to
`PERSONAL_LEARN_PLAN.md`. The default path is the linear order in `README.md` ┬з"How to learn";
your job is to **tailor** it to what the learner already knows and what they're aiming at тАФ
skip/skim what they have, deepen (ЁЯФ┤) what their goal demands, and never violate a prereq.

## Step 1 тАФ Calibrate (short interview)

Ask a compact set of questions (batch them, don't drip one at a time). Keep it to what
changes the plan:

- **Goal** тАФ a target role/JD (paste text or URL), a specific capstone (ЁЯЕ░ Trellix Kanban ┬╖
  ЁЯЕ▒ Pulse Chat), "cover everything", or a topic focus (e.g. "just the React half").
- **Background** тАФ years/comfort per area on a 1тАУ5 scale: TypeScript, React, CSS,
  Node/backend, databases/SQL, GraphQL, auth, realtime, testing, ops/deploy.
- **Time budget** тАФ hours/week and any deadline.
- **Default lane** тАФ ЁЯЯв app ┬╖ ЁЯЯб balanced ┬╖ ЁЯФ┤ deep (per-module overrides allowed).

If `$ARGUMENTS` already carries a goal or a JD, use it and only ask for the missing pieces.

## Step 2 тАФ Pull real signals (don't trust self-report alone)

- Read `PROGRESS.md` for existing status/confidence/notes (schema in `check-progress`). A row
  already `done` with a solid confidence тЖТ candidate to **skip**.
- Detect real completion the cheap way: for any module the learner claims done, grep its
  `src/` for stub markers (`TODO` / `throw new Error("TODO")` / `NotImplemented` / `@stub`,
  excluding `*.test.*`, `*.stories.*`, `dist/`). Still-throwing stub тЗТ not actually done тЗТ
  keep it in the plan. (Same signal-over-vibes rule as `check-progress` тАФ surface drift.)
- If a JD was given, invoke the **`jd-gap-analysis`** skill and use its ranked gaps to raise
  priority/lane on gap modules and lower it on well-covered ones.

## Step 3 тАФ Score each module

For every lesson (all 44: 00тАУ29 + companions), assign a **verdict**:

- **skip** тАФ already solid (high self-rating **and** тАФ if attempted тАФ no stubs / tests pass),
  and not central to the goal.
- **skim** тАФ knows the concept but not this repo's take; read the README, spot-check tasks.
- **do** тАФ the core path: work the tasks at the default lane.
- **deep** ЁЯФ┤ тАФ goal-critical or explicitly wants mastery: take the ЁЯФ┤/companion.

Pick a **lane per module** (goal + rating override the default). Companions (`NNx`) are
optional: include only when the verdict is deep or the goal pulls them (a JD asking for
"custom state management" тЗТ include `13b`; "microfrontends" тЗТ `28c`).

## Step 4 тАФ Order, respecting the prereq graph

Never place a module before its prerequisites. Use the graph in `CURRICULUM.md`
(┬з prereq graph) / `README.md` ┬з"How to learn":

- `00` + `01` gate everything тЖТ always first (even if skimmed).
- Foundations `02 03 04` тЖТ then the **frontend track** `05 (+05b/05c) 06 07 08 09 10 11` and
  **client state/data** `12 13 14`, and the **backend track** `04тЖТ15 16 17 18 19 20`.
- Converge on cross-cutting: `21` Auth (needs `18`+`16`+`01`) тЖТ `22` Realtime (needs
  `18`+`10`) тЖТ `23 24 25` Next (need `05тАУ10`+`16`) тЖТ `26` Testing тЖТ `27` Ops (needs `15/16`+
  `18/23`) тЖТ `28` Perf тЖТ `29` Capstone last.
- If the goal is one capstone, **front-load its modules**: ЁЯЕ░ pulls `11 14 16 20 21(Auth.js)
23тАУ25`; ЁЯЕ▒ pulls `12 15 17тАУ19 21(JWT/Passport) 22`. Keep the other capstone's exclusives as
  skim/optional.

Cluster the ordered list into **phases** (Foundations тЖТ Frontend тЖТ State/Data тЖТ Backend тЖТ
Cross-cutting тЖТ Capstone) and attach a rough time estimate per phase from the time budget.

## Step 6 тАФ Reconcile, don't clobber

`PERSONAL_LEARN_PLAN.md` is the **only** file this skill writes. Never touch `PROGRESS.md`,
module files, or `solution/`. On a re-run, regenerate the plan from fresh signals and note
what changed since the last `Generated` date. Point the learner at `/progress` to track
execution and back here to re-plan.

## Rules

- Respect the chosen lane; only raise to ЁЯФ┤ when the goal or an explicit "master this" asks.
- Every in-scope module appears once with verdict + lane + reason; companions only when pulled.
