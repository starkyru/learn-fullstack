Testing strategy

Which test, at which level, for which confidence — unit, component, integration, E2E, plus MSW and contract tests. The course's dedicated lesson is module 26; the shared harness is packages/testing. Every module already ships tests you run with pnpm --filter ./modules/NN test (and pnpm grade <id> to grade your own src/).

What to mock — boundaries only

Mock the true external edges: network, filesystem, clock, randomness, the browser, an LLM. Do not mock the unit under test or the dependency that holds the logic you're verifying — that's a tautology that passes by construction and catches no regression. Inject the clock/entropy instead of stubbing Date.now. 

// good: real code, injected clock, network mocked at the boundary (MSW)
server.use(http.get("/api/cards", () => HttpResponse.json([{ id: "1" }])));
const result = await loadCards({ now: () => 1_700_000_000 });  // clock injected
expect(result).toEqual([{ id: "1" }]);   // exact shape, discriminating

// bad: mocking the function you claim to test, then asserting the mock back

Assertions must discriminate: assert the exact output/shape, not truthiness or length. A test must fail if the behavior it advertises regresses.

TDD, snapshots, and flake

  • TDD — red → green → refactor. Write the failing test that pins the behavior, make it pass minimally, then clean up under a green bar.
  • Snapshot vs assertion — snapshots catch "did anything change" (good for serialized output/markup); explicit assertions state intent. Prefer assertions for logic; keep snapshots small and reviewed.
  • Kill flake — no real timers/network/order-dependence. Inject the clock, mock the boundary, isolate DB state per test (Testcontainers), await async settle.

Choosing — rules of thumb

  • Pure function/logic → unit. Cheap, keep lots.
  • "Does this component do the right thing when clicked" → component (RTL), query by role/text, not implementation details.
  • "Does the API actually talk to the DB correctly" → integration against a real Postgres (Testcontainers), not a mock.
  • "Can a user complete signup→create→logout" → a few Playwright E2E journeys, not hundreds.
  • Component/integration needs HTTP → MSW at the boundary, one handler set shared across tests.
