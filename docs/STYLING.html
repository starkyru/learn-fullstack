Styling: Tailwind vs CSS Modules vs CSS-in-JS

Companion to module 11, task 5. The headless useDataTable hook is styled once each way — the behavior is identical, only the skin changes. Pick per constraint, not fashion. Siblings: THEMING.html, RESPONSIVE.html.

Same button, three skins

The rendered result is the same; the source differs:

Tailwind (utility)

<button
  class="bg-brand
    text-brand-fg
    rounded-lg px-4 py-2
    font-semibold">
  Save
</button>

CSS Modules

/* Button.module.css */
.btn {
  background: var(--brand);
  border-radius: 8px;
  padding: .5rem 1rem;
}
/* → className={s.btn} */

CSS-in-JS (runtime)

const Btn = styled.button`
  background: ${t =>
    t.colors.brand};
  border-radius: 8px;
  padding: .5rem 1rem;
`;

Themingtokens → preset → CSS varsCSS vars / :roottheme object via ThemeProvider

Runtime costzero (static classes)zero (plain CSS)non-zero: serialize+inject on render (RSC-hostile)

When to use which

  • Tailwind — default for app UI and a shared library skinned per app. Static classes → no runtime, purge keeps the sheet tiny, and a token→preset→CSS-var pipeline (module 11 task 2) themes light/dark by swapping variables. Cost: verbose markup — extract repetition into components, not @apply soup.
  • CSS Modules — when you want plain CSS with local scoping and zero runtime, or a design that's more "stylesheet" than "utilities." Great for a second app that shouldn't inherit the first app's Tailwind config. Cost: prop-driven variance is manual.
  • CSS-in-JS (runtime) — only when styles are genuinely dynamic from props and colocated logic wins. Beware React Server Components: runtime CSS-in-JS needs a client boundary and adds per-render work. Prefer zero-runtime (vanilla-extract, or Tailwind) under RSC.

Rule of thumb

Headless logic (a hook like useDataTable) should never care which of these renders it. Keep behavior in hooks; keep styling swappable. If switching skins forces a logic change, the seam is in the wrong place.
