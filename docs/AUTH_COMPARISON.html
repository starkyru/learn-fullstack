Auth approaches compared

Session cookies vs JWT vs OAuth/OIDC vs Auth.js vs Passport vs magic-link — how each works, what it costs, and when to pick it. The course teaches this by using different stacks on purpose: the Kanban capstone runs Auth.js sessions, the Chat capstone runs JWT + Passport. Modules 21 (compare + OWASP hardening), 21b (hand-rolled sessions, Lucia-style), 25 (Auth.js in Next). Shared code: packages/auth.

Compare, don't crown. There's no single "best" — the right answer is a function of client type, revocation needs, team, and threat model.

Session cookies vs JWT — the core fork

A session is an opaque id in an httpOnly cookie; the server holds the truth in a store, so logout = delete the row (instant revocation). A JWT is a signed, self-contained claim the server verifies without a lookup — great for scale and stateless services, but you can't un-issue an access token, so you keep it short-lived and pair it with a longer refresh token. 

// JWT: short access + rotating refresh
access  = sign(claims, { expiresIn: "15m" })   // in memory, sent as Bearer
refresh = sign({ sub }, { expiresIn: "7d" })   // httpOnly cookie, single-use

// on refresh: verify → issue new pair → invalidate old refresh id.
// if an already-used refresh id comes back → REUSE DETECTED → revoke the family.

Refresh-token reuse detection (module 21): rotate refresh tokens and remember the last valid id per family. A replay of a spent token means it leaked — kill the whole family and force re-login.

CSRF vs XSS — which threat, which defense

  • Cookie auth → CSRF risk. The browser attaches the cookie to forged cross-site requests. Defend with SameSite=Lax/Strict, a double-submit CSRF token, and origin checks (module 21b builds the double-submit from scratch).
  • Token-in-JS → XSS risk. A script injection reads a token from localStorage. Defend by not storing it there + output escaping + CSP. httpOnly cookies sidestep this entirely.

Password hashing

Never store plaintext or fast hashes (MD5/SHA). Use a slow, salted KDF:

AlgorithmNotes
argon2idCurrent best default — memory-hard, resists GPU/ASIC cracking. Preferred.
bcryptBattle-tested, everywhere, 72-byte input cap. Fine if argon2 isn't available.

Authorization: RBAC vs ABAC

  • RBAC — role → permissions (admin, member). Simple, coarse. A Nest guard checks the role (module 21).
  • ABAC — decide on attributes (owner of the resource, org, time, IP). Fine-grained, more logic. Use when "can edit this row" depends on data, not just role.

OAuth2 / OIDC in one breath

OAuth2 authorizes (delegated access via tokens); OIDC adds an identity layer (an id_token = who the user is). Use the Authorization Code flow + PKCE for web/mobile. You end up exchanging the provider's tokens for your own session or JWT — OAuth is the front door, not your whole auth system.

Choosing — rules of thumb

  • One first-party web app, server-rendered → session cookies (or Auth.js on Next).
  • Public/mobile/multi-service API → JWT access + rotating refresh in httpOnly cookies.
  • "Sign in with Google" / third-party data → OAuth2/OIDC, then mint your own session.
  • NestJS with several login methods → Passport strategies behind one guard.
  • Consumer, passwordless → magic link (single-use, expiring).
