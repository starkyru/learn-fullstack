Responsive Design

The principles behind layouts that work on any screen — mobile-first direction, breakpoint strategy, intrinsic (content-out) design, fluid scales, and unit choices — with live, resizable demos. Companion to modules/05d-responsive-design (pure CSS + Tailwind) and modules/05b-css-layout (container queries + clamp() mechanics).

1. Responsive vs adaptive vs fluid

Three words people blur together:

Term What it means Tell
Fluid Sizes flex continuously with the container (%, fr, vw, clamp()). No jumps — smooth at every width.
Adaptive A few fixed layouts that snap at breakpoints (e.g. 3 hand-tuned designs). Discrete jumps; gaps between breakpoints.
Responsive The umbrella: fluid base + breakpoints where the design needs to re-flow. Both — smooth, with a few deliberate snaps.

Modern best practice: go fluid + intrinsic first, and add a breakpoint only where the content actually breaks — not on device sizes.

2. Mobile-first is a direction

Write the smallest screen first as your base styles, then layer up with min-width. The alternative — desktop-first with max-width — walks the cascade backwards and tends to pile up overrides. 

/* mobile-first: base is the phone, one min-width layer adds the sidebar */
.shell { grid-template-columns: 1fr; }              /* base */
@media (min-width: 1024px) {                        /* layer up */
  .shell { grid-template-columns: 20rem 1fr; }
}

3. Intrinsic layout — reflow with zero media queries

The most powerful responsive tool isn't a breakpoint. repeat(auto-fit, minmax(min, 1fr)) fits as many columns of at least min as the container allows and stretches them to fill; when one can't fit, the grid drops a column by itself. 4. Container queries > viewport queries

A media query asks "how wide is the screen?" A container query asks "how wide is my parent?" — so a component is responsive wherever you drop it. Below, the same card list snaps 1 → 2 → 3 columns based on the box width, not the window.

5. Fluid type & space — clamp() + scales

clamp(min, preferred, max) gives type a floor and ceiling with smooth flex between. Pair it with a modular scale (multiply a base by a ratio per step) so headings and spacing grow in tune. Drag the box — the heading scales with the container (cqi units), clamped between 1rem and 2.5rem.

Fluid heading

Units matter. Size in rem, not px, so the scale honors the user's root font-size and browser zoom — the accessibility principle behind pxToRem() and modularScale() in module 05d.

UnitUse for
remType, spacing — zoom-safe, respects root font-size.
% / frFluid track/column widths inside a layout.
chLine length (max-inline-size: 65ch for readable prose).
svh/lvh/dvhFull-height sections on mobile without the URL-bar jump (vh lies).
cqiSizing relative to a query container, not the viewport.

6. Breakpoint strategy

  • Content-driven, not device-driven. Resize until the layout looks bad; put a breakpoint there. Chasing exact phone/tablet widths is a losing game.
  • Fewer is better. Intrinsic layout (§3) removes most breakpoints. Many designs need only one or two real snaps.
  • Name by size, not device. sm/md/lg/xl (module 05d's scale: 640/768/1024/1280) — never .phone/.ipad.
  • Don't forget the meta tag. <meta name="viewport" content="width=device-width, initial-scale=1"> — without it, mobile browsers render at a fake 980px and none of this works.

<div class="grid
  grid-cols-1
  sm:grid-cols-2
  lg:grid-cols-3">

Tailwind's sm:/lg: prefixes are mobile-first min-width queries — an unprefixed class is the phone base. Same cascade, terser syntax. That mapping is responsiveGridClasses() in the module.

8. Checklist

  • Base styles are the smallest screen; min-width only.
  • Reach for auto-fit/minmax and flex-wrap before a media query.
  • Component responsiveness → container queries, not viewport.
  • Type/space on a rem modular scale; clamp() for fluid headings.
  • Line length capped (~65ch); tap targets ≥ 44px; test at 200% zoom.
  • Viewport meta present; full-height uses dvh, not vh.
