Theming & Design Tokens

How UIs get themed — tokens, tiers, the swap mechanisms, and the library landscape. Taught end-to-end in modules/05b (hand-rolled) and modules/11 (tokens → Tailwind + tiers + DTCG build). Siblings: RESPONSIVE.html, STYLING.html.

Token tiers — the part most people skip

A design token is a named design decision stored as data — color.action, space.4, radius.md — spanning color, spacing, type, radius, shadow, motion, and z-index, not just color. It's a contract: change the token, every consumer moves. Flat tokens don't scale, so real systems layer three tiers so retheming is one edit:

Primitive — the raw palette, only place a hex lives: indigo-600 = #4f46e5
Semantic — the role, what it means: action = indigo-600
Component — the slot: button-bg = action

A component references button-bg; dark mode / a partner brand / high-contrast is editing the semantic mapping — primitives and components don't move. This aliasing is what resolveToken() (module 11, task 7) builds by hand.

Modes vs themes: a mode (light / dark / high-contrast) swaps the value a semantic token resolves to; a theme/brand swaps the primitive palette the semantic layer points at. Both are "change one tier, keep the rest."

DTCG — the interchange format

The W3C DTCG JSON is what Figma / Tokens Studio export and Style Dictionary consumes. A token is { "$value", "$type" }; a value can alias another by dotted path: 

{ "color": {
    "blue":   { "600": { "$value": "#4f46e5", "$type": "color" } },
    "action": { "$value": "{color.blue.600}", "$type": "color" }
} }

A build flattens + resolves aliases and emits per-platform output (CSS vars for web). Module 11 task 8 hand-rolls that (🔴 no style-dictionary).

Two mechanisms swap a theme

Everything in the landscape below is one of these under the hood:

  1. CSS custom properties (runtime, zero-JS). Semantic tokens are --action: … on a root; a themed root ([data-theme], prefers-color-scheme) overrides the value. Components use var(--action), so the theme flips with no re-render and no rebuild — the demo above, and what 05b/11 teach.
  2. JS theme object + Context (runtime CSS-in-JS). A ThemeProvider holds a JS object; styled components read it. Flexible, but ships a runtime and fights React Server Components.

Build-time engines (vanilla-extract, Panda, Tailwind v4) compile a token object down to mechanism #1 — JS-object ergonomics with plain-CSS-variable runtime cost.

The library landscape

Six families, roughly ordered runtime-cheap → batteries-included:

  • Hand-rolled CSS vars + data-theme — no dependency, RSC-safe, tiny; you build the tiers + toggle + no-flash SSR yourself (05b).
  • Tailwind + CSS vars — utilities point at var(--x); one class themes every mode. v4 moves tokens into CSS via @theme (11).
  • Runtime CSS-in-JS (Emotion, styled-components, Stitches) — dynamic, JS-driven; runtime bundle + serialization on render, awkward under RSC. Trending out for new work.
  • Zero-/build-time (vanilla-extract, Panda, Tailwind v4) — author tokens + variants in TS, compile to static CSS + variables; type-safe, no runtime, RSC-friendly.
  • Batteries-included kits (MUI, Chakra, Mantine, Radix Themes) — the theme ships with the components; fast to ship, opinionated to redesign.
  • Variants + tokens (cva + shadcn/ui) — variant/size props → Tailwind classes, colors via CSS vars, and you copy the component source into your repo.

Legibility is half the job

Every foreground/background token pair must clear WCAG contrast (4.5:1 body). Build the check into the pipeline — modules/11b-accessibility-wcag derives contrastRatio() by hand. Prefer OKLCH palettes: perceptually uniform lightness makes tint/shade ramps and dark modes predictable.
