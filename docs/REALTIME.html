WebSockets — the concepts (module 22)

  • Gateway — a Nest @WebSocketGateway (socket.io) handles connect/message/disconnect.
  • Rooms — join a channel (board:42) so a broadcast hits only its members.
  • Auth on the socket — verify the token in the handshake before accepting the connection; don't trust the first message. Order matters: authenticate → then join rooms.
  • Reconnect + backoff — the client retries with exponential backoff and re-subscribes on reconnect; the server tolerates duplicate joins.
  • Client as external store — useSocket bridges the socket into React via useSyncExternalStore (subscribe + getSnapshot), so components re-render on push without prop-drilling.
// useSyncExternalStore bridge (the shape module 22 builds)
useSyncExternalStore(
  (cb) => { socket.on("message", cb); return () => socket.off("message", cb); },
  () => store.getSnapshot(),   // MUST be stable — cache it, don't build a fresh object each call
);

Pitfall: a getSnapshot that returns a new object every call → infinite re-render loop. Return a cached reference and only replace it when data actually changes.

Scaling realtime

  • Sticky sessions or a shared adapter (Redis pub/sub) so a broadcast reaches clients pinned to other server instances.
  • Backpressure — drop/coalesce on slow consumers; don't unboundedly buffer.
  • Presence/heartbeat — ping/pong to detect dead connections and reap them.

Choosing — rules of thumb

  • Two-way, low-latency (chat, cursors, multiplayer) → WebSocket.
  • One-way stream (notifications, live logs, progress bars, LLM token streaming) → SSE — simpler, auto-reconnect, no special infra.
  • Already all-in on GraphQL and want typed pub/sub → subscriptions.
  • Updates every few seconds are fine → polling (or TanStack Query refetchInterval). Don't over-engineer.
